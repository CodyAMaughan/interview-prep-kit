# Prep Plan — Foundry (Backend Software Engineer)

## Constraints
- Days until interview: **10**
- Rounds: **recruiter screen (30m) → technical screen (60m: coding problem + past-work discussion) → onsite (coding + system design + behavioral)**

## Priority queue (do in this order)
| # | Action | Skill to run | Maps to | Must-do? |
|---|---|---|---|---|
| 1 | **Gather real numbers** for every project (checkout rebuild, perf work, the bug incident, testing adoption, ops tooling, migration). Fill the ⚠️ flags with actual metrics — never invent. | (offline dig) + prep-intake update | all rounds, esp. "measurable impact" | ✅ |
| 2 | Build 5–6 STAR stories from real work, each landing on a concrete decision + a *found* number | build-stories | recruiter + technical past-work + onsite behavioral | ✅ |
| 3 | Close the backend/API depth gap — be able to walk one real service/API + a SQL schema/query you owned | build-curriculum (backend + SQL tracks) | technical screen + onsite coding | ✅ |
| 4 | Company brief + interviewer playbooks + JD reverse-map talking points | research-prep | all rounds | ✅ |
| 5 | System design fundamentals for the onsite (the only real learn-from-scratch track) | build-curriculum (system-design track) | onsite system design | ⬜ high-ROI |
| 6 | Timed mock technical screen (coding + past-work) | mock-interview | technical screen | ⬜ high-ROI |
| 7 | Morning-of glance sheet (pitch, 2 facts, 3 questions, namedrop lines) | cheatsheets | recruiter + onsite | ⬜ |

## Day-by-day cadence
- **Day 1 (today):** The single biggest must-do — **hunt down the numbers.** Pull analytics/dashboards, PR history, incident tickets for the checkout rebuild, the slow-page perf work, the failed-orders bug, and testing-coverage adoption. This unblocks both stories and the JD's "measurable impact" gap. If a number truly can't be found, keep it flagged and prepare an honest "let me get you that" line — do not fabricate.
- **Day 2:** build-stories — draft the checkout-rebuild, the bug-with-recovery, and the testing/onboarding-leadership stories, slotting in whatever numbers Day 1 surfaced.
- **Day 3:** build-stories — legacy-migration (ambiguity/trade-off) and product/design collaboration stories; build the JD → story map. Rehearse each aloud to 90s.
- **Day 4:** build-curriculum — backend/API track: write up one real service/API you owned (design, data model, trade-offs) so the "backend-leaning" requirement has proof.
- **Day 5:** build-curriculum — SQL track: drill schema + query problems closed-book (technical screen has live coding).
- **Day 6:** research-prep — Foundry brief (small-business tools), why-this-company angle, interviewer playbooks, JD reverse-map.
- **Day 7:** build-curriculum — system-design track: fundamentals + one practice design relevant to "small-business product backend."
- **Day 8:** mock-interview — timed technical screen (coding + past-work discussion); have coach grade. Fix the gaps it finds.
- **Day 9:** Spaced repetition — re-rehearse all stories to 90s, re-run weak SQL/system-design items, tighten the backend/API walkthrough.
- **Day 10 / Morning-of:** cheatsheets glance sheet — 90s pitch, 2 Foundry facts, 3 smart questions, namedrop lines mapping your experience to their small-business-tools problems.

## "Done" looks like
- You can state a **real number** for the checkout rebuild, the perf work, the bug incident, and the testing adoption — or you have a crisp, honest "I'd want to confirm the exact figure, but the direction was X" with the dig already attempted. **No invented metrics, ever.**
- You can defend every "strong" JD line cold and walk one backend service/API + one SQL schema/query you actually built.
- Each of 5–6 STAR stories delivered in ≤90s, "I" not "we", landing on a decision + a found number + a one-line lesson.
- You can run a basic system design out loud for a small-business product backend.
- You can say *why Foundry* in their language (small-business tools) and have 3 sharp questions ready.
- The honest framing is internalized: this is sharpening proven skills and surfacing proof — the only fresh study is system design.
